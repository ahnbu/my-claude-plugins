# Hook 세션 ID 확보 — 멀티세션 충돌 문제 및 대책

> 저장일: 2026-03-05 | 원본: C:\Users\ahnbu\.claude\plans\hidden-hopping-moth.md

## 발단: 사용자 요청

> "동일 폴더에서 멀티 세션을 하는 경우가 빈번하다. 그러면 의미가 없어지는 것 아닌가? 이에 대한 대책은?"

현재 구현(`capture-session-id.js`)이 `<cwd>/.claude/.current-session-id` 단일 파일에 세션 ID를 기록하는 방식이므로, 같은 CWD에서 여러 세션이 동시 실행되면 마지막 시작 세션이 파일을 덮어써서 세션 ID가 오염된다는 문제 제기.

---

## Context

my-session-wrap의 `capture-session-id.js`가 SessionStart hook에서 `<cwd>/.claude/.current-session-id`에 세션 ID를 기록하는 방식은 **동일 CWD 멀티세션에서 마지막 시작 세션이 파일을 덮어쓰는** 치명적 결함이 있다. 이를 해결할 방안을 수립한다.

---

## 문제 분석

```
세션 A 시작 → .current-session-id = "aaa-111"
세션 B 시작 (같은 CWD) → .current-session-id = "bbb-222" (덮어씀)
세션 A에서 /wrap 실행 → cat .current-session-id → "bbb-222" ← 잘못된 ID!
```

단일 파일 방식은 동일 CWD 멀티세션에서 세션 격리가 불가능하다.

---

## 해결 후보 비교

| 방안 | 원리 | 세션 격리 | 복잡도 | 비고 |
|------|------|:---------:|:------:|------|
| **A. CLAUDE_ENV_FILE** | hook이 per-session env 파일에 `export CLAUDE_SESSION_ID=xxx` 기록 → Bash에서 `$CLAUDE_SESSION_ID`로 접근 | ✅ 완벽 | 낮음 | Claude Code 내장 메커니즘. **Node.js에서 미검증** |
| B. 세션별 파일명 | `.session-id-<session_id>` 형태로 저장 | ❌ 순환참조 | - | wrap이 자기 session_id를 알아야 파일을 찾을 수 있음 → 본말전도 |
| C. UserPromptSubmit + /wrap 필터 | `/wrap` 입력 시에만 session_id 저장 | ⚠️ 경합 | 중간 | 두 세션이 동시 /wrap 시 경합 가능 |
| D. JSONL 파일 탐색 | /wrap 시점에 JSONL에서 세션 ID 추출 | ⚠️ 불확실 | 높음 | 멀티세션 시 어느 JSONL이 "이 세션"인지 특정 어려움 |

### 방안 A가 최적인 이유

- **CLAUDE_ENV_FILE은 Claude Code가 세션별로 생성하는 임시 파일**. 각 세션이 독립된 env 파일을 가지므로 멀티세션 충돌이 원천 차단됨
- hook에서 `process.env.CLAUDE_ENV_FILE`로 경로를 얻고, `export CLAUDE_SESSION_ID=xxx`를 기록하면, 해당 세션의 Bash 도구에서 `$CLAUDE_SESSION_ID`로 접근 가능
- **이전 "실패" 판정은 오류**: bash 스크립트가 Windows cmd.exe에서 실행되지 않은 것이지, CLAUDE_ENV_FILE 자체가 비어있었던 것이 아닐 수 있음. Node.js 환경에서는 한 번도 테스트하지 않았음

### Phase 2: CLAUDE_ENV_FILE 기반 구현 (Phase 1 성공 시)

**`capture-session-id.js` 프로덕션 코드**:
```js
const fs = require("fs");
const path = require("path");

let input = "";
process.stdin.on("data", (chunk) => { input += chunk; });
process.stdin.on("end", () => {
  if (!input) return;
  try {
    const data = JSON.parse(input);
    const { session_id, cwd } = data;
    if (!session_id) return;

    // 1차: CLAUDE_ENV_FILE (per-session 환경변수) — 멀티세션 안전
    const envFile = process.env.CLAUDE_ENV_FILE;
    if (envFile) {
      fs.appendFileSync(envFile, `export CLAUDE_SESSION_ID=${session_id}\n`);
    }

    // 2차: 파일 기반 (단일세션 fallback)
    if (cwd) {
      const dest = path.join(cwd, ".claude", ".current-session-id");
      fs.mkdirSync(path.dirname(dest), { recursive: true });
      fs.writeFileSync(dest, session_id);
    }
  } catch (_) {}
});
```

### Phase 3 — 방안 재분석 (2026-03-05)

**새로운 결론**: Bash 도구에서 세션 ID를 직접 얻는 내장 메커니즘은 없음.

#### 검토 옵션

| 옵션 | 원리 | 안전성 | 결론 |
|------|------|:------:|------|
| CLAUDE_ENV_FILE | hook→Bash 환경변수 | ❌ Bash 도구 미전달 | **기각** |
| JSONL 최신 파일명 | `/wrap` 실행 시 가장 최근 .jsonl = 현재 세션 | ⚠️ 동시 응답 시 경합 | **채택 (실용적)** |
| `.current-session-id` 파일 | SessionStart 덮어쓰기 | ❌ 멀티세션 충돌 | **fallback만 유지** |
| UserPromptSubmit `/wrap` 감지 | `/wrap` 입력 시 해당 세션 ID 저장 | ⚠️ 동시 /wrap 경합 | **보조 수단** |

#### 채택: JSONL 최신 파일명 방식 (Primary)

`/wrap` 실행 시점에 AI가 직접 조회:

```bash
# <cwd>에 해당하는 project 폴더의 가장 최근 JSONL 파일명 = 현재 세션 ID
ls -t ~/.claude/projects/$(pwd | sed 's|[/\\]|--|g' | sed 's|^C:||' | sed 's|^--|C--|')/*.jsonl \
  2>/dev/null | head -1 | xargs -I{} basename {} .jsonl
```

**왜 실용적인가**:
- `/wrap` 실행 = AI 응답 중 = 현재 세션의 JSONL이 지금 막 쓰이고 있음
- 다른 세션이 같은 순간 응답할 확률은 극히 낮음 (동시 /wrap 시나리오)
- 단일세션에서는 100% 정확

**한계**: 동일 CWD에서 두 세션이 동시에 AI 응답을 받는 경우 경합 가능 (현실적으로 드묾)

### Phase 5 — Hook stdout 방식 (2026-03-05) ← 최종 해결

> 사용자 지적: "나는 보통 멀티세션 5~6개를 띄워놓는데, 그러면 그 중 하나가 wrap을 실행할 때, 그게 올바른 세션이라는 것을 어떻게 보장하지?"

**핵심 발견**: Hook stdout(console.log)은 **해당 세션의 AI 컨텍스트에 시스템 메시지(system-reminder)로 주입**된다.
- 세션 A의 hook stdout → 세션 A의 AI에만 보임
- 세션 B의 hook stdout → 세션 B의 AI에만 보임
- **파일 공유 없음, 경합 없음, 완벽한 세션 격리**

검증된 사례: `inject-plugin-guidelines.js`의 `console.log`가 `SessionStart hook success: ⚠️ [플러그인 개발 폴더 감지]...`로 표시됨.

|------|------|------|
| **유지** | SessionStart 파일 기록을 안전망으로 남겨두기 | ❌ 기각 |
| **제거** | SessionStart 분기 + 관련 imports 삭제, hooks.json에서 SessionStart capture hook 제거 | ✅ 채택 |

**제거를 선택한 근거**:

"파일 fallback이 의미 있는 에지케이스가 있는가?"를 검토한 결과:

- `/wrap`은 항상 사용자 프롬프트로 실행됨 → UserPromptSubmit hook이 항상 먼저 발동
- 두 hook이 **동일한 `capture-session-id.js`를 실행** → UPS가 실패하는 상황에서 SS만 성공하는 경우가 존재하지 않음
- 플러그인 미로드 / hook 크래시 시: 두 hook 모두 실패 → 파일도 없음
- SKILL.md 없이 `/wrap` 수동 실행: 세션 ID 획득 절차 자체가 없음

**결론**: 실질적인 에지케이스 없음. 파일 기록 로직은 pure dead code.

**변경 내용**:
- `capture-session-id.js`: SessionStart 분기 + `fs`/`path` import 제거 → UPS `console.log`만 남김
- `hooks.json`: SessionStart의 `capture-session-id.js` hook 엔트리 제거
- 버전: `2.8.0` → `2.8.1`
