### Context

3개 스킬(session-dashboard, session-timeline, session-transcript)이 동일한 JSONL 세션 데이터를 독립적으로 읽고 처리 중:

| 현재 문제 | 영향 |
|-----------|------|
| build.js가 매번 전체 스캔 → 44MB HTML 생성 | 대시보드 빌드 느림 |
| timeline/transcript가 매번 DFS로 파일 탐색 | 단일 세션 조회에 전체 디렉토리 순회 |
| `.build-cache.json`(42MB)이 build.js 전용 | 다른 스킬과 공유 불가 |
| `stripSystemTags`, `getTextContent` 등 중복 구현 | 유지보수 부담 |

**목표**: 하나의 SQLite DB로 통합하여 모든 스킬이 공유. 대시보드는 DB 조회, timeline/transcript도 DB에서 추출.

### 기술 선택: `node:sqlite` (Node.js 내장)

| 기준 | `node:sqlite` | `better-sqlite3` | JSON 파일 |
|------|:---:|:---:|:---:|
| 외부 의존성 | 없음 | node-gyp 필요 | 없음 |
| Windows 빌드 이슈 | 없음 | VS Build Tools 필요 | 없음 |
| 인덱스 기반 단건 조회 | O(1) | O(1) | 전체 로드 |
| 증분 upsert | 행 단위 | 행 단위 | 전체 rewrite |
| 동시 접근 (WAL) | 지원 | 지원 | 파일 락 충돌 |

- Node.js v24.13.1에서 `DatabaseSync`, `json_extract` 정상 동작 확인
- Experimental 경고가 뜨지만 핵심 API(`DatabaseSync`, `prepare`, `run`, `get`, `all`)는 안정적
- **폴백**: API 변경 시 `better-sqlite3` 어댑터로 교체 가능 (인터페이스 래핑)

### DB 스키마

```sql
-- 세션 메타데이터 (현재 allMeta 배열에 대응)
CREATE TABLE sessions (
  session_id TEXT PRIMARY KEY,
  type TEXT NOT NULL DEFAULT 'session',     -- 'session'|'plan'|'codex'
  title TEXT, keywords TEXT,                -- JSON 배열
  timestamp TEXT NOT NULL, last_timestamp TEXT,
  project TEXT, git_branch TEXT,
  models TEXT,                              -- JSON 배열
  message_count INTEGER, tool_use_count INTEGER,
  total_input_tokens INTEGER, total_output_tokens INTEGER,
  tool_names TEXT,                          -- JSON 객체 {name: count}
  first_message TEXT, file_path TEXT,
  mtime REAL,                               -- 증분 빌드용
  -- plan 전용
  slug TEXT, is_completed INTEGER, char_count INTEGER, linked_session_id TEXT,
  -- codex 전용
  originator TEXT
);

-- 대시보드용 메시지 (sessionsData에 대응)
CREATE TABLE messages (
  session_id TEXT NOT NULL, seq INTEGER NOT NULL,
  role TEXT NOT NULL, text TEXT, timestamp TEXT,
  tools TEXT,                               -- JSON [{name, input}]
  PRIMARY KEY (session_id, seq)
);

-- plan 원문
CREATE TABLE plan_contents (
  session_id TEXT PRIMARY KEY, content TEXT NOT NULL
);

-- timeline/transcript용 정규화된 이벤트
CREATE TABLE events (
  session_id TEXT NOT NULL,
  seq INTEGER NOT NULL,
  agent_id TEXT DEFAULT '',                 -- '' = main, 'agent-xxx' = subagent
  kind TEXT NOT NULL,                       -- 'user_text'|'tool_use'|'tool_result' 등
  source TEXT DEFAULT 'main',
  timestamp TEXT, timestamp_ms INTEGER,
  data TEXT,                                -- JSON blob (kind별 구조)
  PRIMARY KEY (session_id, agent_id, seq)
);

CREATE INDEX idx_sessions_timestamp ON sessions(timestamp);
CREATE INDEX idx_events_session ON events(session_id);
```

## 2. 검토 이슈 및 결론

### 2-1. 새 JSONL/Plan 생성 시 자동 반영되는가?

**결론: 아니요. Pull 방식.**

원본 plan은 `sync()` (전체 증분) / `syncSingleSession()` (단건)을 **호출 시점에** 실행하는 구조. 파일 워처나 훅 기반 자동 트리거 없음.

**실제 호출 진입점 (JSONL을 실제 파싱하는 6개):**

| 진입점 | 범위 | 라이브러리 |
|--------|------|-----------|
| `/ss` → build.js | 전체 스캔 (2,500+) | 자체 파싱 + mtime 캐시 |
| session-timeline 스킬 | 단건 (sessionId) | lib/session/ |
| session-transcript 스킬 | 단건 (sessionId) | lib/session/ |
| history-insight 스킬 | 프로젝트 단위 | bash/AI 직접 |
| session-analyzer 스킬 | 단건 | bash/AI 직접 |
| /continue (sessionId 포함 시) | 단건 | AI 직접 Read |

나머지 훅 진입점(capture-session-id, check-handoff)은 JSONL을 파싱하지 않음.

### 2-3. DB 동기화 전략 비교

| 기준 | A. Pull-only | B. Stop 훅 트리거 | C. Hybrid (권장) |
|------|:---:|:---:|:---:|
| 방식 | 조회 시 sync() | Stop 훅에서 현재 세션 즉시 upsert | Stop 훅 + 조회 시 증분 sync |
| 구현 복잡도 | ✅ 낮음 | 보통 | ⚠️ 두 경로 |
| `/ss` 빌드 속도 | ⚠️ 첫 호출 느림 | ████░░ 대부분 DB에 존재 | ✅ ██████ 미반영분만 증분 |
| 외부 세션 반영 | ✅ 전체 스캔 보장 | ❌ 훅 미실행 시 누락 | ✅ 조회 시 보정 |
| 훅 실패 영향 | ✅ 해당 없음 | ⚠️ DB 누락 (JSONL 원본 무사) | ✅ 조회 시 보정 |

**Stop 훅 빈도 부담?**
→ Stop 훅은 매 assistant 턴마다 실행되지만, `syncSingleSession()`은 mtime 비교 후 변경 없으면 즉시 return (수십ms). fs.watch 데몬도 동일 빈도이므로 차이 없음.

---

### 2-4. fs.watch 데몬 방식은 유효한가?

**결론: 원리적으로 유효하나, 운영 복잡도 대비 이득이 크지 않음.**

- 기술적으로는 가능 (SessionStart 훅에서 백그라운드 spawn)
- 다중 세션 중복 spawn 관리, 프로세스 고아화 위험 존재
- Windows에서 `fs.watch` 안정성 이슈 (중복 이벤트, 락 충돌)
- Stop 훅이 동일 효과를 더 단순하게 달성

---

### 2-5. DB 도입 시 timeline/transcript 스크립트가 필요한가?

**현재 스크립트가 하는 일:**

```
JSONL 로드(1) → 정규화(2) → 변환/계산(3) → 출력 포맷팅(4)
```

DB가 (1)+(2)를 대체. (3)+(4)는 여전히 필요:

- **timeline-builder**: 턴 간 duration 계산, tool 대기시간 집계, 에이전트별 그룹핑 (계산 로직)
- **transcript-builder**: role별 포맷팅, tool_use↔tool_result 매칭, 마크다운 출력 (포맷팅 로직)

**스킬에서 DB 직접 조회 가능 여부:**

| 기준 | 스크립트 유지 | 스킬에서 DB 직접 조회 |
|------|:---:|:---:|
| 출력 일관성 | ✅ 결정적 | ⚠️ AI 해석마다 미세 차이 |
| 토큰 비용 | ✅ 스크립트가 처리, AI는 결과만 수신 | ⚠️ raw events를 컨텍스트에 로드 |
| 유연성 | ❌ 출력 변경 시 코드 수정 | ✅ 프롬프트만 수정 |
| 대형 세션 (500+ events) | ✅ 스크립트가 집계 | ❌ 컨텍스트 초과 위험 |

**결론:**
- timeline: 계산 로직이 핵심 → **스크립트 유지, 데이터 소스만 DB로 교체**
- transcript: 경량 세션은 DB 직접 조회 가능, 대형 세션은 스크립트가 잘라서 전달

### 2-6. DB vs 현재 build-cache.json 비교

| 기준 | 현재 (.build-cache.json, 42MB) | SQLite DB |
|------|:---:|:---:|
| 증분 업데이트 | ⚠️ 전체 JSON rewrite | ✅ 행 단위 INSERT OR REPLACE |
| 메모리 사용 | ❌ 42MB 전체 로드 | ✅ 쿼리 결과만 로드 |
| 단건 조회 | ❌ 전체 로드 후 필터 | ✅ 인덱스 O(1) |
| 다른 스킬과 공유 | ❌ build.js 전용 포맷 | ✅ 표준 SQL 인터페이스 |
| 동시 접근 | ❌ 파일 락 충돌 위험 | ✅ WAL 모드 병행 |
| 디버깅 | ⚠️ 42MB JSON 열기 어려움 | ✅ sqlite3 CLI로 즉시 쿼리 |
| 구현 난이도 | ✅ 이미 동작 중 | ⚠️ 마이그레이션 작업 필요 |

**핵심 판단**: build.js 단독으로는 현재 캐시로 충분. DB의 진짜 가치는 6개 진입점이 공유하는 통합 데이터 레이어.

### 2-7. SQLite DB 기반으로 현재 index.html 구현 가능한가?

**현재 구조:**
```
build.js → 2,500 세션 JSON 직렬화 → index.html에 인라인 임베딩
         ├── window.__SESSIONS_META__ = [...] (세션 목록)
         └── window.__SESSIONS_DATA__ = {...} (세션별 메시지)
         → 결과: 44MB 단일 HTML (순수 클라이언트 사이드)
```

**DB 전환 방식 3가지:**

| 방식 | 설명 | index.html 변경 | 결과 크기 |
|------|------|:---:|:---:|
| **A. 현재 유지 (권장)** | build.js가 DB 조회 → 동일하게 HTML 임베딩 | ✅ 없음 | ❌ 44MB 그대로 |
| **B. serve.js + REST API** | API 엔드포인트 추가, fetch() 기반 조회 | ⚠️ 전면 개작 | ✅ 경량화 가능 |
| **C. sql.js (브라우저 SQLite)** | DB 파일을 브라우저에 로드 | ⚠️ 중간 수준 | ❌ DB 전체 다운로드 |

**결론**: **방식 A가 현실적.** DB 전환 자체는 index.html에 영향 없이 가능. 44MB HTML 경량화는 별도 scope(방식 B).
