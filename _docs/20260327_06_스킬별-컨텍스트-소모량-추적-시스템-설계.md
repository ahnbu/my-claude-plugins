# 스킬별 컨텍스트 소모량 추적 시스템 설계

## 발단: 사용자 요청

> "현재 context 소모량을 기록하는 로직을 활용해서, 스킬들이 사용되었을 때, 컨텍스트가 얼마나 소모되는지 기록하고 추적, 평가하고 싶다."

기존 statusline이 `context_window.used_percentage`를 실시간으로 수신하고 있으므로, 이를 활용해 Skill 호출 전후 cp% 델타를 실측하는 시스템 설계.

## 작업 상세내역

### 1단계: 기존 인프라 탐색

탐색 결과 확인한 기존 인프라:

| 구성요소 | 기록 방식 | 저장 위치 |
|----------|-----------|-----------|
| statusline | cp >= 80% 시 JSON 기록 | `~/.claude/scripts/context-warning/<sid>.json` |
| 세션 DB | `total_input/output_tokens`, `skill_calls` 집계 | `sessions.db` |
| session-parser.js | JSONL 파싱 시 `usage` 객체에서 토큰 집계 | DB 인제스트 시 |

실시간 Skill 감지 훅은 없음. `skill_calls` 컬럼은 있으나 세션 단위 집계이고 시점 정보 없음.

### 2단계: 접근법 논쟁 — 추정 vs. 실측

초기 plan에서 제안된 방식:
- **tool_result 콘텐츠 크기** + **전후 input_tokens 델타** 조합

사용자 문제 제기:
> "이 방식은 실제로 컨텍스트 소모량이 아니라, 추정 아닌가?"

두 측정 시스템 비교:

| 구분 | continue 스킬 (기존) | 제안 방식 (신규) |
|------|---------------------|-----------------|
| 단위 | 세션 전체 | 스킬 호출 1회 |
| 측정값 | 전체 cp% | 스킬 전후 cp% 델타 또는 토큰 수 |
| 질문 | "세션이 얼마나 찼나?" | "이 스킬이 얼마나 먹었나?" |
| 역할 | context limit 감지 → 이어가기 대상 식별 | 스킬별 비용 비교·최적화 |

→ 두 시스템은 **경쟁 관계가 아닌 보완 관계**. 측정 대상 자체가 다름.

논의 결론 (/feed-my 피드백 결과 포함):

feed-my 피드백 요약 (**보류 ⚠️**):

> cp% 델타 방식은 생각보다 정확하지 않다. 제안한 tool_result 크기 방식도 "추정"이 맞지만, 오히려 cp% 델타보다 더 안정적인 추정이다.

| 방식 | 얻는 것 | 잃는 것 |
|------|---------|---------|
| **cp% 델타** (statusline 전후 비교) | "실제 컨텍스트 사용률" 느낌 | 측정 타이밍 불일치, 노이즈, turn 단위 한계 |
| **tool_result 크기** (JSONL 파싱) | 안정적·재현 가능한 추정, 스킬 호출 시점 정확 | chars→tokens 환산 계수 필요 |

**cp% 방식의 핵심 한계**:
- statusline 갱신은 **turn 단위**지 tool_use 단위가 아님. 한 turn에 스킬이 여러 번 호출되면 before/after 분리 불가.
- 노이즈 평균 상쇄는 두 방식 모두 동일 — 차별점이 아님.
- tool_result 방식이 스킬 호출 시점을 정확히 특정할 수 있어 더 안정적.

**최종 채택** (피드백 반영 후): cp% 델타 1차 + tool_result 교차검증 2차 구조는 유지. 단, cp% 방식의 한계(turn 단위) 인지하고 집계 시 노이즈 처리 필요.

### 3단계: 시스템 설계 및 구현

설계된 데이터 흐름:

```
[PostToolUse hook — Skill 감지]
  → skill-context/<session_id>.jsonl에 append:
    {"type": "skill", "skill": "wrap", "ts": "...", "session_id": "..."}

[Statusline — 매 청크마다]
  → skill-context/<session_id>.jsonl에 append:
    {"type": "cp", "ts": "...", "cp": 52, "session_id": "..."}

[분석 스크립트]
  → JSONL → skill 이벤트 전후 가장 가까운 cp 레코드 매칭
  → delta = cp_after - cp_before → 집계 리포트
```

구현된 파일:

| 파일 | 변경 내용 |
|------|-----------|
| `my-session-wrap/hooks/skill-context-logger.js` | **신규** — PostToolUse hook, Skill 감지 + JSONL 기록 |
| `my-session-wrap/hooks/hooks.json` | PostToolUse에 skill-context-logger.js 추가 |
| `~/.claude/scripts/cc-alchemy-custom.mjs` | skill-context 활성 세션에 cp% append (existsSync 조건부) |

**성능 최적화**: `existsSync(scFile)` 체크로 Skill 미사용 세션에는 I/O 비용 없음.

### 5단계: cp_before null 버그 발견 및 수정

문제: 세션 첫 번째 스킬 호출 직전에 cp 레코드가 없으면 `cp_before = null` → delta 계산 포기.

## 리스크 및 미해결 이슈

- **cp% 해상도 한계**: 1% 단위 측정이므로 경량 스킬(예: `wrap` 0.8%)은 0으로 반올림될 수 있음. 실제 값이 소수점 이하인 경우 통계적 노이즈 처리 필요.
- **cp% turn 단위 한계**: statusline은 tool_use 단위가 아닌 turn 단위로 갱신. 한 turn에 스킬 여러 번 호출 시 before/after 분리 불가능. 집계 시 보정 로직 필요.
- **TTL 정리 미구현**: `skill-context/*.jsonl` 3일 TTL 정리 로직이 설계에만 있고 구현 미완.
- **세션 DB 연동 미결**: 수일 데이터 축적 후 DB 저장 여부 미결정.

## Context

스킬(Skill) 호출이 세션 컨텍스트 예산에 미치는 영향을 **실측**하고 싶다. 현재 statusline이 매 assistant 청크마다 `context_window.used_percentage`를 수신하므로, Skill 호출 전후 cp% 델타를 직접 측정할 수 있다.

## 설계 원칙

- **1차 측정**: cp% 델타 (statusline 실측)
- **2차 교차검증** (향후): tool_result 콘텐츠 크기 (JSONL 파싱)
- **매핑 키**: 두 데이터를 연결하기 위해 `session_id` + `skill_name` + `ts` (타임스탬프)를 공통 키로 사용

## 데이터 흐름

```
[PostToolUse hook — Skill 감지]
  → skill-context/<session_id>.jsonl에 append:
    {"skill": "wrap", "ts": "2026-03-27T10:00:01Z", "phase": "after"}

[Statusline — 매 청크마다]
  → skill-context/<session_id>.jsonl에 append:
    {"ts": "2026-03-27T10:00:01.5Z", "cp": 52, "source": "statusline"}

[후처리 분석 스크립트]
  → JSONL 읽기 → skill 이벤트 전후 가장 가까운 cp% 레코드 매칭
  → 델타 계산 → 리포트 출력

[향후 교차검증 — session-parser.js]
  → JSONL에서 Skill tool_result 콘텐츠 크기 추출
  → 매핑 키: session_id + skill_name + ts (±5s 허용)
```

### Step 2. Statusline — cp% 상시 기록

**파일**: `C:\Users\ahnbu\.claude\scripts\cc-alchemy-custom.mjs` (수정)

현재 (486-508행):
```js
const cp = data.context_window?.used_percentage || 0;
if (cp >= 80 && data.session_id) {
  // context-warning에 기록
}
```

추가: cp >= 80 블록과 **별도로**, skill-context 디렉토리에 활성 세션이 있을 때만 cp% 기록:

```js
// skill-context 활성 세션이면 cp% 추적
const scDir = join(dirname(SCRIPT_PATH), "skill-context");
const scFile = join(scDir, `${data.session_id}.jsonl`);
if (data.session_id && existsSync(scFile)) {
  appendFileSync(scFile, JSON.stringify({
    type: "cp", ts: new Date().toISOString(), cp, session_id: data.session_id
  }) + "\n");
}
```

**핵심**: `existsSync(scFile)` 체크로 skill-context JSONL이 존재하는 세션(= Skill 호출이 1회 이상 발생한 세션)에서만 cp%를 기록. 스킬 미사용 세션에는 I/O 비용 없음.

### Step 3. 분석 스크립트

**파일**: `my-session-wrap/skills/continue/scripts/analyze-skill-context.mjs` (신규)

기능:
1. `~/.claude/scripts/skill-context/*.jsonl` 읽기
2. 각 `type: "skill"` 이벤트에 대해:
   - 직전 `type: "cp"` 레코드 → `cp_before`
   - 직후 `type: "cp"` 레코드 → `cp_after`
   - `delta = cp_after - cp_before`
3. 집계 출력:
   ```
   스킬 컨텍스트 소모 분석
   ─────────────────────────
    스킬명              호출수  평균Δcp%  최대Δcp%
    brainstorming          12    +4.2%    +7.1%
    wrap                   45    +0.8%    +1.2%
    find-skills-my          8    +2.1%    +3.5%
   ```

CLI:
```bash
node analyze-skill-context.mjs                    # 전체 집계
node analyze-skill-context.mjs --skill wrap       # 특정 스킬 상세
node analyze-skill-context.mjs --session <sid>    # 특정 세션 상세
```

## 매핑 키 설계 (교차검증용)

두 데이터 소스를 연결하는 공통 키:

| 필드 | cp% 실측 (JSONL) | tool_result 크기 (향후 parser) |
|------|-------------------|-------------------------------|
| `session_id` | ✅ 파일명 | ✅ 세션 파싱 시 자동 |
| `skill_name` | ✅ `type: "skill"` 레코드 | ✅ `block.input.skill` |
| `ts` | ✅ ISO timestamp | ✅ `entry.timestamp` |

매칭 로직: `session_id` 일치 + `skill_name` 일치 + `|ts_cp - ts_parser| < 5s`

이를 통해 향후 "cp% 델타 4.2% ≈ tool_result 14,700 chars" 같은 환산 계수를 도출할 수 있다.
