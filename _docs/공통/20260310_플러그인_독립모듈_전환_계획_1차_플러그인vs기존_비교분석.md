## ⚠️ 정정: "플러그인 없으면 동작 불가"는 부정확한 표현

이 문서 초안에서 `my-session-id`, `my-session-wrap`에 대해 "hook이 핵심 기능 자체 → 플러그인 없으면 동작 불가"라고 기술했으나, **이는 부정확하다.**

### Hooks가 동작하는 경로 (Plugin 불필요)

| 위치 | 범위 | Plugin 필요? |
|------|------|:---:|
| `~/.claude/settings.json` → `"hooks"` 키 | 전역 (모든 프로젝트) | **아니오** |
| `.claude/settings.json` → `"hooks"` 키 | 해당 프로젝트만 | **아니오** |
| `<plugin>/hooks/hooks.json` | Plugin 활성화 시 | 예 |

**Plugin은 hooks를 배포·공유하기 위한 패키징 래퍼일 뿐**, hooks 기능 자체는 Plugin 없이 `settings.json`의 `"hooks"` 키로 완전히 동작한다.

### `~/.claude/hooks/` 폴더 활용법

스크립트 파일은 `~/.claude/hooks/` 아래 기능별로 자유롭게 정리할 수 있다. `settings.json`의 `command`에서 경로를 지정하면 된다.

```
~/.claude/hooks/
  session-id/
    capture-session-id.js
  session-wrap/
    check-handoff.js
    inject-plugin-guidelines.js
```

```json
// ~/.claude/settings.json
{
  "hooks": {
    "UserPromptSubmit": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "node ~/.claude/hooks/session-id/capture-session-id.js"
      }]
    }],
    "SessionStart": [{
      "matcher": "",
      "hooks": [{
        "type": "command",
        "command": "node ~/.claude/hooks/session-wrap/check-handoff.js"
      }]
    }]
  }
}
```

### `~/.claude/hooks/hooks.json`은 자동 인식되지 않는다

`~/.claude/hooks/hooks.json`이라는 **파일을 Claude Code가 자동으로 읽어가는 메커니즘은 없다.** Plugin의 `<plugin>/hooks/hooks.json`은 Plugin 시스템이 자동 인식하지만, 그 외 경로의 `hooks.json`은 자동 인식 대상이 아니다. hooks 정의는 반드시 `settings.json`의 `"hooks"` 키 안에 해야 한다.

### 시나리오별 권장 방식

| 시나리오 | 권장 방식 |
|----------|-----------|
| **본인만 사용** | `~/.claude/settings.json` + 스크립트 파일 자유 배치 (`~/.claude/hooks/`) |
| **타인에게 배포** | Plugin 구조 (`<plugin>/hooks/hooks.json`) — Plugin 시스템이 자동 등록 |

## Plugin 시스템의 존재 목적과 시나리오별 비교

### Plugin 시스템이 만들어진 이유

공식 문서 원문: *"Plugins let you extend Claude Code with custom functionality that can be **shared across projects and teams**."*

Plugin 시스템은 **공유·배포 활성화**를 위해 설계되었다. 해결하려는 근본 문제 3가지:

| 문제 | Plugin이 해결하는 방식 |
|------|----------------------|
| 팀/커뮤니티 공유 불가 | 마켓플레이스 배포 + `/plugin install` 한 줄 설치 |
| 이름 충돌 | 네임스페이싱 (`/plugin-name:skill-name`) |
| 버전 관리 부재 | `plugin.json` semantic versioning + 자동 업데이트 감지 |

> Plugin의 모든 고유 장점(네임스페이싱, 자동 업데이트, `${PLUGIN_ROOT}`, 마켓플레이스 설치)은 **"남이 쓸 때"를 위한 기능**이다. 본인 전용 도구에는 Plugin 구조가 설계 의도와 맞지 않는다.

### 시나리오별 비교

| 고려요소 | 🅰️ 타인 플러그인 사용 | 🅱️ 내 것을 플러그인으로 | 🅲️ 내 것을 플러그인 없이 |
|----------|:---:|:---:|:---:|
| **초기 구성** | ✅ `/plugin install` 한 줄 | hooks.json + plugin.json + marketplace.json 작성 | settings.json hooks 키 작성 |
| **수정 → 반영** | N/A (소비자) | ❌ push → marketplace update → plugin update | ✅ 파일 수정 = 즉시 반영 |
| **hooks 인식** | ✅ 자동 (hooks.json) | ✅ 자동 (hooks.json) | ✅ 자동 (settings.json hooks 키) |
| **경로 관리** | ✅ `${PLUGIN_ROOT}` 자동 | ❌ 소스 ↔ marketplace ↔ cache 3중 | ✅ 경로 1개 |
| **관리 복잡도** | ██████ 없음 | ░░░░░░ 3중 복사본 동기화 | ✅ ██████ 소스 = 런타임 |
| **디버깅** | ██░░░░ | ██░░░░ marketplace 복사본 추적 | ✅ ██████ 파일 = 실행본 |
| **네임스페이싱** | ✅ 충돌 방지 | 본인만 사용 시 무의미 | 본인만 사용 시 무의미 |
| **배포·공유** | ✅ 이 용도 | ✅ 가능 (하지만 본인용이면 과잉) | ❌ 수동 공유만 |
| **자동 업데이트** | ✅ 유의미 | 본인 코드라 무의미 | N/A |

**비교표 해설:**
- 🅰️와 🅱️에서 "초기 구성"의 차이는 실질적으로 없다 — 개발자 입장에서 어차피 수동 파일 작성이 필요하다. plugin.json/hooks.json을 쓰느냐 settings.json을 쓰느냐의 차이일 뿐.
- `--plugin-dir ./path` 플래그로 marketplace 없이 Plugin을 로드하는 방법도 존재하나, **개발 중 로컬 테스트 전용**이다. 일상 사용은 공식 의도가 아님.

### 결론: 본인 전용 도구의 올바른 구조

```
~/.claude/
  settings.json            ← hooks 정의 (SessionStart, UserPromptSubmit 등)
  hooks/

    session-id/

  skills/
    session-dashboard/

  commands/               ← 필요 시 수동 배치
```

### 전체 플러그인 Hook 상세 분석

#### my-session-wrap (플러그인 유지 권장 — 배포 시)

| Hook | 이벤트 | 기능 | 스킬 대체 가능? |
|---|---|---|---|
| `capture-session-id.js` | UserPromptSubmit | 매 프롬프트마다 세션ID를 AI 컨텍스트에 자동 주입 | ❌ 스킬은 사용자 호출 필요 |
| `check-handoff.js` | SessionStart | 24시간 이내 handoff 파일 감지 → 자동 안내 | ❌ 사용자 호출 없이 실행 |
| `inject-plugin-guidelines.js` | SessionStart | 특정 디렉토리 감지 후 경고 자동 주입 | ❌ 동일 이유 |
| `ensure-commands.js` | SessionStart | 커맨드 자동 배포 | ⚠️ 수동 복사로 대체 가능 |

> 본인만 사용한다면 `~/.claude/settings.json` + `~/.claude/hooks/session-wrap/` 스크립트 배치로 Plugin 없이 동일하게 구현 가능.
