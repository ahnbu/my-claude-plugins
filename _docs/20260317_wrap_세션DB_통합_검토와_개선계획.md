# /wrap 세션DB 활용 검토 및 개선 플랜 (v2)

## Context

- `/wrap` 실행 시 체감 속도 느림 보고
- 세션DB 활용이 성능·속도·토큰 효율 개선에 도움이 되는지 검토 요청
- **v1 결론 "DB 통합 비추천"은 철회.** 아래 3가지 근거로 DB 통합이 필요하다고 판단 수정

## 초기 분석 오류와 수정 근거

### 오류 1: doc-save가 이미 DB 활용 패턴을 확립했음을 무시

doc-save SKILL.md는 DB를 3가지 용도로 활용:
- `find_linked_plan.js <sessionId>` — 연결된 plan 탐색
- `find_plan_versions.js <sessionId>` — plan 변천 이력 복원
- `extract_session_gaps.js <sessionId>` — 세션 대조 검수 (gap 스캔)

**같은 플러그인 내 wrap이 이 패턴을 안 쓰는 것이 비일관적.**

### 오류 2: _docs에 세션DB 토큰 효율 개선 실적이 문서화되어 있음

`_docs/` 문서들에 따르면:
- 빌드 속도 13배 개선 (1분+ → 4.5초)
- 42MB JSON 전체 로드 → 쿼리 결과만 로드
- DB events 테이블이 이미 정규화 → AI가 재파싱 불필요
- 3개 스킬이 동일 DB 인프라 공유 → 중복 파싱 제거

### 오류 3: "AI가 이미 알고 있다"는 위험한 가정

- 긴 세션(50+ 턴)에서 대화 자동 압축 → 초반 의사결정·도구 호출 디테일 소실
- AI는 토큰 수·도구 호출 횟수를 정확히 셀 수 없음
- JSONL 직접 읽기는 최악 — 수천 줄이 컨텍스트에 유입 → 토큰 폭발
- DB의 `events` 테이블은 **사전 파싱된 시간순 정규화 이벤트** 제공

| §2 변경 내역 | ✅ + git diff | ⚡ 보조 | tool_use 이벤트로 "무엇을 했는지" 구조적 확인 |
| §3 피드백 루프 | ✅ | ⚡ 보조 | 도구 대기시간, 에러 패턴 등 병목 자동 탐지 가능 |

- `lib/session/` 코드 삭제/이동 — 이미 session-loader.js가 DB 경로를 상대참조하므로 유지
- handoff 핵심 콘텐츠의 AI 의존 — §4(다음 작업), §5(교훈)는 여전히 AI 컨텍스트 전담

## 검증 방법

1. `extract-session-summary.js`를 현재 세션 ID로 실행 → JSON 출력 확인
2. `/wrap` 실행 → DB 조회 성공 시 handoff 헤더에 토큰·도구 통계 포함 확인
3. `/wrap` 실행 (DB 없는 환경) → graceful degradation으로 기존 방식 동작 확인
4. 긴 세션(50+ 턴) 후 `/wrap` → gaps 탐지로 누락 보완되는지 확인
