## Q2. session-timeline은 무엇이 나오는가? 언제 쓰는가?

### 출력물 해석

타임라인은 **세션 내 모든 도구 호출을 시간순으로 나열한 표**:

```
| # | 시각  | 간격  | 도구 | 입력            | 대기  | 상태  | 결과 요약 |
|---|-------|-------|------|-----------------|-------|-------|-----------|
| 1 | 00:20 | 00:20 | Glob | **/*            | 00:00 | ok    | ...       |
| 2 | 00:21 | 00:01 | Glob | **/SKILL.md     | 00:00 | ok    | ...       |
| 3 | 04:09 | 00:24 | Bash | echo "=== ..."  | 00:00 | ok    | ...       |
```

**상단 요약**: 세션 총 소요시간, AI 턴 소요, 도구 대기시간, 서브에이전트 소요 등

### 이것은 "일정표"가 아니라 "성능 프로파일러"

| 오해 | 실제 |
|------|------|
| 일정표처럼 날짜별 세션 배치 | ❌ 아님 |
| 개별 세션의 도구 호출 시간 분석 | ✅ 맞음 |

### 사용 시나리오

| 시나리오 | 타임라인이 답하는 질문 |
|----------|----------------------|
| "이 세션 왜 이렇게 오래 걸렸지?" | 어떤 도구에서 병목이 발생했는지 (간격 열) |
| "서브에이전트가 너무 오래 돌았나?" | 서브에이전트 총 소요시간 확인 |
| "에러가 몇 번 났지?" | 상태 열에서 error 건수 |
| "AI가 생각하느라 멈춘 구간은?" | 간격이 큰 구간 = AI 사고 시간 |

### 대시보드 통합 시 가치

현재 대시보드에는 **세션별 성능 메트릭이 전혀 없음**. 타임라인의 요약 데이터(세션 소요, AI 턴, 도구 대기)를 대시보드 세션 목록에 표시하면 유용.

### Transcript에서 배울 점 (대시보드 개선 아이디어)

1. **`isMeaningfulToolResult()` 함수**: 도구 결과가 의미있는지 판단하는 로직 — 대시보드의 메시지 뷰에 "결과 숨기기/에러만" 필터 추가 가능
2. **`shortenToolName()`**: MCP 도구명 축약 — 대시보드 도구 표시에 즉시 적용 가능
3. **`getToolContext()`**: 도구 입력에서 핵심 파라미터만 추출 — 대시보드 도구 요약 개선
4. **상대 시간 표시**: `[00:14]` 형식 — 세션 시작 기준 경과시간이 절대시간보다 직관적

### Transcript 결과물 구조 (실제 파일에서)

```markdown
**[00:14] 사용자**
현재 'history-insight" 스킬이 안뜨는데...

---

**[00:19] 어시스턴트**
현재 작업 디렉토리가...

> **도구** `Glob` — **/*
> ```json
> {"pattern":"**/*","path":"..."}
> ```

> 결과 `Glob`
> C:\Users\...\SKILL.md ...
```

→ 도구 호출과 결과가 **인라인으로 쌍을 이루어** 표시됨. 대시보드는 이 구조를 가지고 있지 않음.

## Q4. Timeline 데이터 — DB 통합 vs 온디맨드 버튼

### 비교

| 항목 | DB 프리빌드 (빌드 시 계산) | 온디맨드 버튼 (클릭 시 실행) |
|------|--------------------------|---------------------------|
| 응답 속도 | ✅ 즉시 (이미 계산됨) | 1~3초 대기 |
| 저장 공간 | 세션당 ~2KB 추가 | ✅ 추가 없음 |
| 데이터 신선도 | 빌드 시점 스냅샷 | ✅ 항상 최신 |
| 구현 복잡도 | 중간 (DB 스키마 확장 + 빌드 수정) | ✅ 낮음 (버튼 + API 호출) |
| 전체 세션 비교 | ✅ 가능 (모든 세션 메트릭 비교) | ❌ 개별 세션만 |
| 빌드 시간 영향 | 세션당 0.1~0.3s 추가 | ✅ 없음 |

### 권장: **하이브리드** — 요약은 DB, 상세는 온디맨드

1. **DB에 요약 메트릭만 저장** (빌드 시):
   - `session_elapsed_ms`, `total_turn_duration_ms`, `total_tool_wait_ms`, `tool_count`, `error_count`
   - 세션 목록에서 "35min / 도구 82건 / 에러 3건" 표시 가능
   - 세션 간 성능 비교, 정렬, 필터링에 활용

2. **상세 타임라인은 "타임라인 보기" 버튼**으로 온디맨드:
   - 클릭 시 `session_timeline.js` 실행 → 결과를 모달/패널에 표시
   - 모든 세션에 대해 프리빌드할 필요 없음 (관심 세션만 확인)

## 최종 권장안 요약

| 스킬 | 방향 | 근거 |
|------|------|------|
| **session-analyzer** | 보류 (코드 유지, 배포 안 함) | 디버그 로그 기반 → DB 전환 필요. 현재 사용 시나리오 없음. 필요 시 재설계 |
| **session-timeline** | 대시보드 하이브리드 통합 | 요약 메트릭 → DB/세션 목록에 표시. 상세 → 온디맨드 버튼 |
| **session-transcript** | 대시보드 메시지 뷰 개선에 기법 흡수 | transcript-builder의 필터링/매칭/축약 로직을 대시보드에 적용. 스킬 자체는 폐기 |
| **lib/session/** | 유지 | timeline-builder, transcript-builder의 함수들을 대시보드가 직접 import |

### 실행 계획

**Phase 1: 대시보드 메시지 뷰 개선** (transcript 기법 흡수)
- `isMeaningfulToolResult()`, `shortenToolName()`, `getToolContext()` → 대시보드 적용
- 도구 결과 필터 드롭다운 추가 ("전체/에러만/경로만/숨김")
- 상대 시간 표시 옵션

**Phase 2: 세션 성능 메트릭 추가** (timeline 요약 통합)
- DB 스키마에 `session_elapsed_ms`, `tool_count`, `error_count` 등 추가
- 세션 목록에 성능 배지 표시
- "타임라인 보기" 버튼 → 온디맨드 상세 패널

**Phase 3: 정리**
- session-transcript 스킬 디렉토리 삭제 (기법은 대시보드에 흡수 완료)
- session-timeline 스킬 디렉토리 삭제 (온디맨드 실행은 lib/ 직접 호출)
- session-analyzer는 그대로 보류 (코드만 유지)
- 테스트(session-tools.test.js) 유지

### 검증

- `/ss`로 대시보드 열고 → 메시지 뷰에서 도구 결과 필터링 동작 확인
- 세션 목록에 성능 메트릭 표시 확인
- "타임라인 보기" 버튼 클릭 → 상세 패널 확인
- `node --test tests/session-tools.test.js` 통과
