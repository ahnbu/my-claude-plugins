## 작업 상세내역

### 1단계: 변경 요소별 증상 연관성 분석

| 변경 요소 | 변경 내용 | 현재 증상 | 원인 유형 |
|-----------|-----------|-----------|-----------|
| **Phase 1** Gemini 캐시 | `session-db.js`: sentinel 레코드로 재파싱 방지 | 없음 | ✅ 무관 |

### 2단계: HTML 33MB 구성 분석

"데이터 분리로 99MB→33MB 감소" 주장을 실측 검증:

| 섹션 | 크기 | 비율 |
|------|------|------|
| UI 코드 (HTML/CSS/JS) | 62KB | 0.2% |
| `__SESSIONS_META__` | 2.1MB | 6% |
| **`__SEARCH_INDEX__`** | **31MB** | **93%** |

**핵심 발견**: 33MB의 정체는 검색인덱스(31MB)이며, 데이터 분리와 무관. 이전 세션의 커밋 메시지("99MB→8.3MB")도 사실과 다름 (실제 33MB).

### 3단계: 과최적화 판단 (/feed-my)

| | 기존 (인라인 99MB) | 현재 (분리 33MB) | 실질 차이 |
|---|---|---|---|
| 초기 로딩 파싱 | ❌ 99MB JSON 파싱 | 33MB JSON 파싱 | 66MB 절감이지만 31MB 잔존 |
| 세션 클릭 속도 | ✅ 즉시 (0ms, 인메모리) | HTTP fetch 필요 | ❌ 오히려 악화 |
| 서버 의존성 | ✅ 없음 (file:// 가능) | ❌ serve.js 필수 상주 | ❌ 신규 장애점 추가 |
| 코드 복잡도 | ✅ 단순 (파일 1개, 서버 없음) | ❌ fetch, 캐시, 서버, 증분빌드, 가상스크롤 | ❌ 유지보수 부담 대폭 증가 |

**결론**: 초기 로딩 1~2초 줄이려고, 매번 세션 클릭할 때마다 지연을 만들고, 서버 의존성을 추가하고, 버그 3개를 만듦. 과최적화 비추천 ❌.

### 4단계: Gemini 중복 UUID "신규 17" 원인 분석 및 수정

원복 후에도 빌드 시 "Gemini: 신규 17"이 매번 반복 표시되는 문제를 추가 조사했다.

**원인**: 동일 UUID(예: `3d0d4219`)의 Gemini 세션 파일이 2~3개 존재. `_upsertSession()`의 `INSERT OR REPLACE`가 PRIMARY KEY(`session_id`)로 덮어쓰기하여 **마지막 파일의 file_path만** DB에 남음. 이전 파일들은 DB 레코드가 없으므로 매 빌드마다 "NEW (no DB record)"로 반복 감지.

| UUID | 파일 수 | DB 레코드 | 결과 |
|------|---------|-----------|------|
| `3d0d4219` | 3개 | 세션 1개 (마지막 파일만) | 앞의 2개 매빌드 NEW |
| `8e09941c` 등 7개 | 각 2개 | 세션 1개 (마지막 파일만) | 앞의 1개 매빌드 NEW |

**수정**: `_syncGeminiDir()`에서 `_upsertSession` 호출 전, 동일 `session_id`로 DB에 이미 레코드가 있는지 확인. 있으면 현재 파일을 `gemini_excluded` sentinel로 저장하여 다음 빌드에서 캐시 히트.

**검증**: 빌드 2회 실행 → 1회차에서 9개 중복 파일 sentinel 생성, 2회차에서 Gemini 신규 0, 캐시 99 확인.
