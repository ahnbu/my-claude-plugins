### Phase 1: `doc` 커맨드 구현 (`query-sessions.js`)

**파일**: `~/.claude/my-claude-plugins/shared/query-sessions.js`

추가된 구성요소:

| 함수 | 역할 |
|------|------|
| `syncSingleMessages(db, sessionId, filePath, type)` | JSONL 재파싱 → messages 테이블 upsert. Claude/Codex/Gemini 지원. Antigravity·Plan은 DB 캐시 사용 |
| `messagesToMarkdown(metaRow, messages, detailed)` | role별 헤더 + 텍스트 + tool 요약 → 마크다운 포맷 |
| `cmdDoc(db, sessionId, opts)` | on-demand sync → messages 조회 → 마크다운 반환 |

**옵션 처리**:
- `--detailed`: tool input JSON 포함 출력
- `--no-sync`: JSONL 재파싱 생략 (DB 캐시 사용)
- `--limit <N>`: 최근 N개 메시지만 출력 (기본: 전체)

**on-demand sync 소요**: ~0.7s (events force sync 2.1s 대비 3배 빠름)

**진행중 세션 조회 동작**: `doc` 실행 시 JSONL 파일 직접 재파싱 → 마지막 완료 턴까지 반영. 스트리밍 중인 응답은 JSONL에 미기록이므로 미포함.

**다른 세션에서 진행중 세션 조회**: 가능. `syncSingleMessages`가 대상 세션의 JSONL 파일을 직접 읽음. read-only라 write 충돌 없음.

## 의사결정 기록

### `--detailed` · `--no-sync` CLI 유지 vs 제거

feed-my 검토를 두 단계로 진행:

**1차 검토 (규칙 파일 제거 여부)**:

| 옵션 | 규칙 파일 제거 이유 | 결정 |
|------|-------------------|------|
| `--no-sync` | AI가 "완료 세션인지" 판단 불가 → stale 데이터 위험. 0.7s 절감은 체감 불가 | ✅ 제거 |
| `--detailed` | AI 자의 판단으로 verbose 남발 위험. 사용자 명시 요청 시만 필요 | ✅ 제거 |

**2차 검토 (CLI 코드 자체 삭제 여부)**:

| 옵션 | CLI 유지 근거 | 결정 |
|------|-------------|------|
| `--no-sync` | 다른 세션에서 진행중 세션 조회 시 JSONL 쓰기 중 → sync 충돌 회피 실제 use case 존재 | ✅ CLI 유지 |
| `--detailed` | 사람이 직접 디버깅 시 tool 인자 확인 필요 — 규칙 제거(AI 자동사용 방지)와 CLI 제거(기능 삭제)는 별개 문제 | ✅ CLI 유지 |

**최종 결론**: 규칙 파일에서는 제거(AI 자동 사용 방지), CLI 코드·`--help`에는 유지(수동 사용 가능). `--help` examples에서 `--detailed` 예시 제거(편향 방지).

### 옵션 문구 설계 원칙

규칙 파일 옵션 표기는 "설명"이 아닌 "트리거 조건"이어야 AI가 판단 가능:

| 나쁜 예 | 좋은 예 |
|--------|--------|
| `--scope <타입 필터>` | `--scope <claude\|codex\|plan\|gemini\|antigravity>`로 AI 소스 필터 |
| `--limit <N>` | `--limit <N>`으로 반환 세션 수 제한 (기본 10) |
| 옵션 단순 나열 | 커맨드·옵션 분리, 적용 커맨드 명시 |

## Phase 1: `query-sessions.js`에 `doc` 커맨드 추가

### 목표

`node query-sessions.js doc <session_id> [--detailed]`로 세션 대화 내용을 마크다운으로 출력.

### 설계

- **데이터 소스**: messages 테이블 (buildDocText와 동일 범위)
- **기본 출력**: user 텍스트 + assistant 텍스트 + 도구명 요약 (~31KB)
- **`--detailed`**: + tool input JSON 포함 (~94KB)
- **제외**: tool_result (messages에서 NULL), thinking (DB에 없음)

### 핵심 문제: 진행중 세션의 messages stale

`sync()`는 대시보드 빌드 시에만 실행. 진행중 세션에서 `doc` 커맨드 호출 시 messages가 오래됨.

**해결**: `doc` 커맨드 실행 시 해당 세션의 messages를 on-demand 적재.

```
processSession(file_path) → messages 파싱 (16ms)
_upsertMessages(session_id, messages) → DB INSERT (0.7s)
getMessages(session_id) → 조회 (2ms)
```

총 소요: ~0.7s. `syncSingleSession`의 events force sync (2.1s)보다 3배 빠름.

### 옵션

| 옵션 | 기본값 | 설명 |
|------|--------|------|
| `--detailed` | false | tool input JSON 포함 |
| `--limit N` | 전체 | 최근 N개 메시지만 |
| `--no-sync` | false | messages sync 생략 |
