# 세션 대시보드 로딩 성능 최적화

## 발단: 사용자 요청

세션 대시보드(`/ss`) 로딩에 약 10초 소요. 사용자 체감 목표: 3초 이내.
실측 기반 분석으로 3개 근본 원인(Gemini 캐시 미스, 99MB 인라인 데이터, 렌더링 최적화 부재)을 식별하고, 3-Phase 개선 계획을 수립.

## 작업 상세내역

### 진단 결과

| 항목 | 수치 |
|------|------|
| 출력 HTML 크기 | **99MB** (97.9%가 `__SESSIONS_DATA__` 인라인 JSON) |
| 세션 수 | 2,083개 |
| 메시지 수 | 117,349개 |
| DB 크기 | 231MB |
| HTML 중 앱 코드 | 59KB (0.06%) |
| gzip 압축 시 | 99MB → 17MB (83% 감소) |

### 근본 원인 분석

1. **Gemini 캐시 미스**: 필터(auto-call, <60초)에 걸리는 세션이 DB에 미저장 → `_findSessionByFilePath`가 항상 null → 매번 JSON 재파싱 후 폐기. "신규 17"로 인해 `totalNew > 0` → 99MB HTML 전체 재빌드.
2. **99MB 인라인 데이터**: 117K 메시지를 `<script>` 블록에 JSON 임베드 → 브라우저 파싱 2-5초 + JSON→JS 변환 1-3초 + 메모리 200-400MB.
3. **렌더링 최적화 부재**: 2,083개 세션 전체 DOM 렌더링, 검색 시 대화 내용 전체 스캔, 이벤트 위임 없음.

### 3-Phase 개선 계획

- **Phase 1**: Gemini 캐시 버그 수정 — 필터된 세션도 `type='gemini_excluded'`로 DB에 sentinel 레코드 저장, 디버그 로그 추가
- **Phase 2**: 데이터 분리 — 메타(2MB)만 인라인, 메시지는 세션별 JSON 파일로 분리하여 lazy load
- **Phase 3**: Progressive Rendering — 가상 스크롤, 검색 인덱스 사전 구축, 이벤트 위임, 메시지 렌더링 최적화

### 논의 후 추가된 의사결정

#### 전문 검색 손실 문제 (Phase 2 부작용)

Phase 2에서 `loadConversation()`이 async가 되면서 `getFilteredSessions()` 내 동기 검색 불가 → `__SEARCH_INDEX__`(메타 필드만)로 대체 → 전문 검색 불가.

- **결정**: 빌드 시 검색 인덱스에 대화 텍스트(첫 2,000자/세션) 포함
- **근거**: 인덱스 크기 ~4MB 추가 (HTML 3.1MB→~7MB), 99MB 대비 93% 감소. 검색 속도 <200ms 유지, 초기 로드 <1초 유지.

#### "더보기" 50개 제한 제거

- **결정**: 50개 초기 제한 + "더보기" 버튼 제거, 전체 메시지 렌더링 복원
- **근거**: 매번 클릭하는 것은 핵심 목표(빠른 열람)에 위배

#### data/*.json 증분 생성

현재: `totalNew > 0`이면 2,085개 JSON 전부 재생성 (~4-6초 I/O). 새 세션 1개 추가에 불필요한 전체 I/O.

- **결정**: data/ 폴더를 캐시 레이어로 활용한 증분 생성
- **방식**: DB sessionId 목록 vs data/ 파일 목록 비교 → 없으면 생성, mtime 다르면 갱신, DB에 없으면 삭제(고아 파일 정리)
- **트레이드오프**: 실질적으로 없음

#### Gemini 동일 UUID 복수 파일 (잔존 "신규 17")

동일 세션 UUID의 타임스탬프별 복수 파일 → `_upsertSession`이 PRIMARY KEY(sessionId)로 마지막 파일만 저장 → 이전 파일의 `file_path` 매치 실패 → 매번 "신규". 이번 작업 범위 밖.

- 결정: 3-Phase 순차 적용 (Gemini 캐시 수정 → 데이터 분리 → Progressive Rendering)

빌드 로그: `Gemini: 신규 17, 캐시 40` — 사용자가 Gemini를 열지 않았는데도 매번 17개가 "신규"로 처리됨.

Gemini 17 신규 → totalNew > 0 (build.js:41)
→ 2,083개 세션 전체 DB 쿼리 (build.js:54-60)
→ 99MB HTML 재생성 (build.js:63-74)

**원인 분석 (`session-db.js:358-386`):**
Gemini 캐시 로직은 `file_path` 기반 조회(`_findSessionByFilePath`) + `mtime` 비교로 동작.
가능한 실패 지점:
- (A) Gemini CLI가 백그라운드로 세션 파일을 수정하여 mtime 변경
- (B) 필터(auto-call, <60초)에 걸려 DB에 저장되지 않는 세션이 매번 재파싱
  - `processGeminiSession()` 호출 → 필터에서 `continue` → DB에 미저장 → 다음 빌드에서 또 파싱

**(B)가 유력**: 필터에 걸리는 세션은 DB에 저장되지 않으므로 `_findSessionByFilePath`가 항상 null 반환 → 매번 JSON 파싱 후 폐기. 이 세션들은 `countCb`를 호출하지 않으므로 "신규 17"에는 포함되지 않지만 CPU를 낭비함. "신규 17"은 실제로 데이터가 변경된 파일이거나 mtime이 바뀐 파일.

**실행 시 확인 방법**: `_syncGeminiDir`에 디버그 로그를 추가하여 각 파일의 absPath, existing 여부, mtime 비교 결과를 출력.

#### 원인 2: 99MB 인라인 데이터 → 브라우저 파싱 병목 (Critical)

모든 세션의 모든 메시지(117K개)를 `<script>` 블록에 JSON으로 임베드.
- HTML 파싱: ~2-5초 (99MB 텍스트)
- JSON → JS 객체 변환: ~1-3초 (메인 스레드 블로킹)
- 메모리: ~200-400MB 점유

#### 원인 3: 렌더링 최적화 부재 (Medium)

- 세션 목록: 2,083개 전체를 `.map()` → `innerHTML`로 한번에 렌더링
- 검색: `getFilteredSessions()`에서 `loadConversation()` 호출 → 대화 내용 전체 스캔

### Phase 1: Gemini 캐시 버그 수정 (즉시 효과: 재빌드 방지)

**파일**: `shared/session-db.js` — `_syncGeminiDir` 메서드

**변경 1-1: 필터된 세션도 DB에 기록하여 재파싱 방지**

현재 문제: 자동호출 세션(auto-call), 60초 미만 세션은 `continue`로 건너뛰어 DB에 저장되지 않음 → 매번 JSON 파싱 후 폐기 반복.

```javascript
// 현재 (session-db.js:370-382)
const result = processGeminiSession(absPath, projectRoot);
if (!result || result.metadata.userEntryCount === 0) continue;  // DB 미저장 → 다음에 또 파싱
if (/\((codex|claude)\)/i.test(result.metadata.firstMessage)) continue;  // DB 미저장
if (durationSec < 60) continue;  // DB 미저장
this._upsertSession(result.metadata, mtime);  // 여기까지 와야 저장

// 개선: 필터된 세션도 mtime만 기록 (sentinel 레코드)
```

구현 방향: 필터 조건에 걸리는 세션도 `sessions` 테이블에 `type='gemini_excluded'`로 저장하고 mtime을 기록. 이렇게 하면:
- `_findSessionByFilePath` → 기존 레코드 발견
- `mtime` 비교 → 일치하면 캐시 히트
- 메시지는 저장하지 않음 (공간 절약)
- `getAllMeta()`에서 `type='gemini_excluded'` 제외

**변경 1-2: 디버그 로그 추가 (원인 확정용)**

`_syncGeminiDir`에 verbose 모드 로그를 추가하여 각 파일의 캐시 히트/미스 이유를 출력:
```
[gemini] session-abc.json: existing=null → NEW (no DB record)
[gemini] session-def.json: mtime 1710000000 vs db 1709999000 → CHANGED
[gemini] session-ghi.json: cached ✓
```

**변경 1-3: `totalNew` 계산에서 Gemini 임계값 적용**

`build.js:41`에서 Gemini 신규가 소수(예: 5개 미만)일 때는 메타데이터만 갱신하고 HTML 재빌드를 생략하는 옵션. (Phase 2의 아키텍처 변경 후에는 불필요할 수 있으므로 선택적)

**예상 효과**: 변경 없는 빌드에서 99MB HTML 재생성 방지 → 빌드 시간 수십 초 → <1초

### Phase 2: 데이터 분리 — 메타만 인라인, 메시지는 Lazy Load (핵심)

**파일**: `build.js`, `index.html`

현재: `__SESSIONS_META__` (2MB) + `__SESSIONS_DATA__` (97MB) = 99MB 인라인
목표: 메타(2MB)만 인라인, 메시지는 세션 선택 시 개별 로드

**변경 2-1: build.js — 메시지를 개별 JSON 파일로 분리**

```javascript
// 메타만 인라인
const metaJson = JSON.stringify(allMeta).replace(/<\//g, "<\\/");

// 메시지는 세션별 JSON 파일로 출력
const dataDir = path.join(DIST_DIR, "data");
fs.mkdirSync(dataDir, { recursive: true });
for (const meta of allMeta) {
  const data = meta.type === "plan"
    ? db.getPlanContent(meta.sessionId)
    : db.getMessages(meta.sessionId);
  const safeId = meta.sessionId.replace(/[^a-zA-Z0-9_:-]/g, "_");
  fs.writeFileSync(path.join(dataDir, `${safeId}.json`), JSON.stringify(data));
}
```

**변경 2-2: index.html — Lazy Loading 구현**

```javascript
// 세션 선택 시 메시지를 fetch로 로드
async function loadConversation(sessionId) {
  if (messageCache.has(sessionId)) return messageCache.get(sessionId);
  const safeId = sessionId.replace(/[^a-zA-Z0-9_:-]/g, "_");
  const resp = await fetch(`data/${safeId}.json`);
  const data = await resp.json();
  messageCache.set(sessionId, data);
  return data;
}
```

**변경 2-3: serve.js 수정 — data/ 디렉토리 정적 서빙**

`serve.js`의 HTTP 핸들러에 `data/*.json` 경로 추가.

**예상 효과**:
- HTML 크기: 99MB → **~2MB** (메타만)
- 초기 로드: ~10초 → **<1초** (2MB 파싱)
- 메시지 로드: 세션 선택 시 개별 JSON fetch (~50-500KB)
- file:// 프로토콜에서는 fetch가 동작하지 않으므로 serve.js 경유 필수 (현재도 serve.js 사용 중이므로 문제 없음)

**변경 3-1: 세션 목록 가상 스크롤 (Virtual Scrolling)**

```javascript
function renderSessionList() {
  const filtered = getFilteredSessions();
  const container = document.getElementById("sessionList");
  const ITEM_HEIGHT = 72; // 추정 높이
  const VISIBLE_COUNT = Math.ceil(container.clientHeight / ITEM_HEIGHT) + 5;

  container.style.height = (filtered.length * ITEM_HEIGHT) + 'px';
  // 스크롤 위치에 따라 보이는 항목만 렌더링
  const startIdx = Math.floor(container.scrollTop / ITEM_HEIGHT);
  const visibleItems = filtered.slice(startIdx, startIdx + VISIBLE_COUNT);
  // ... 렌더링
}
```

**변경 3-2: 검색 인덱스 사전 구축**

현재: 검색마다 `loadConversation()` → 대화 내용 전체 스캔
개선: 빌드 시 검색 인덱스(메타 필드 + 첫 메시지)를 사전 생성. 대화 내용 전문 검색은 별도 버튼으로 분리.

**변경 3-3: 이벤트 위임**

**예상 효과**:
