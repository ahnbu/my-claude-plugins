# 세션 대시보드 검색 잘림(truncation) 진단과 수정

## 발단: 사용자 요청

사용자가 이전 세션에서 입력한 "방금 git 레포가 아니라고 판단하기 위해서 사용한 과정을 누락없이 답변으로 그대로 제시하라"를 세션 대시보드에서 검색했으나, 해당 세션(`f3d3d431-ab9b-438c-be03-9697755ce5d3`)이 검색 결과에 나오지 않는 문제.

## 작업 상세내역

### 1. 세션 특정

- `query-sessions.js search "git 레포가"`로 검색 → `89bf5a00` 세션 발견 (파생 세션)
- 사용자 지적으로 원본 세션 `f3d3d431` 확인 — JSONL line 431에서 해당 프롬프트 최초 입력
- `89bf5a00`은 이 내용을 복사해서 개선방안 수립 요청으로 재사용한 별도 세션

### 2. 검색 실패 원인 분석

대시보드 검색 로직 (`index.html:1016~1030`) 조사:

```javascript
// 세션 메시지 텍스트를 합산 후 처음 10,000자만 검색 대상으로 사용
const texts = conv.filter(m => m.text).map(m => m.text).join(" ");
fields.push(texts.substring(0, 10000));  // ← 핵심 병목
```

| 지표 | 값 |
|------|-----|
| 세션 f3d3d431 전체 텍스트 | 13,073자 |
| "git 레포가" 최초 등장 위치 | 10,753자 (seq 162) |
| 검색 잘림 한도 | 10,000자 |
| **결과** | 753자 차이로 검색 누락 |

## 의사결정 기록

- 결정: `substring(0, 10000)` 제거 (세션 + plan 모두)
- 근거: 데이터가 이미 `window.__SESSIONS_DATA__`에 전부 로드되어 있으므로 메모리 추가 사용 없음. 19.5MB `includes()` 스캔은 수십ms 수준으로 체감 차이 없음.
- 트레이드오프: 세션 10,000개 이상으로 증가 시 검색 둔화 가능성 있으나, 그 시점에 FTS(WASM SQLite + trigram) 전환 검토.

### 검토하고 제외한 대안

| 대안 | 제외 사유 |
|------|----------|
| C안: null text에 tool_result 채우기 | 설계 의도 (도구 노이즈 제외). DB 크기 급증, HTML 번들 폭증 |
| B안: DB FTS5 즉시 도입 | 대시보드가 self-contained HTML이라 아키텍처 변경 필요. 중기 과제로 보류 |
| substring 한도 증가 (10K→100K) | 근본 해결이 아님. 제거가 동일 비용으로 완전 해소 |
